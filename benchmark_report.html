Notes
  • ADSMUserGeneralDetails uses ReplacingMergeTree ORDER BY (unique_id) in ClickHouse — id is not the primary sort key, so WHERE id > N cursor queries cannot use the sparse primary index and trigger a full scan per page.
  • IN FILTER cursor (10k ids, raw): large IN literal re-parsed per page → MySQL ~370 ms; ClickHouse columnar scan ~95 ms.
  • Temp table opt: 10k IDs loaded once into an indexed table; cursor pages use JOIN — eliminates the IN literal from every page query.
  • CH UPDATE/DELETE pattern: INSERT with same unique_id + newer updated_at; engine deduplicates on next background merge.

ClickHouse FINAL notes
  • FINAL forces the ReplacingMergeTree engine to merge duplicate versions on-the-fly during the query, keeping only the latest row per ORDER BY key (unique_id).
  • WHERE is_deleted = 0 excludes soft-deleted rows that were inserted with the upsert-delete pattern (is_deleted=1 + newer updated_at).
  • ADSMUserAccountDetails and ADSMUserExchangeDetails are plain MergeTree — FINAL is not applied to them and their queries are identical between the two columns.
  • The overhead depends on how many unmerged duplicate versions exist in the table parts. After a full background merge the overhead approaches zero.
